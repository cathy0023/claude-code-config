# Case Study：事故深度复盘

针对反复修复多次才彻底解决的问题，进行系统性的证据收集、根因分析和文档化沉淀。

## 核心理念

1. **AI 自主为主**：大部分证据收集和分析由 AI 自动完成，减少人工负担
2. **反复修复是核心线索**：聚焦"为什么前几次没修对"，这是 case-study 区别于普通 bug 复盘的独特价值
3. **5 Whys 到底**：不满足于表面原因，逐层追问直到系统性根因
4. **产物即交付**：生成的 case-study 文档就是最终交付物，不做额外流程

## 触发条件

此 skill 适用于以下场景：
- 一个 bug 被修了 2 次以上才彻底解决
- 一个事故的修复过程暴露了流程/架构/测试的深层问题
- 需要做团队级别的经验沉淀和最佳实践更新

## 输入

用户提供最小描述即可启动：
- `/case-study 上周三的支付回调故障`
- `/case-study 最近修复的那个 OOM bug`
- `/case-study 从最近 commit 里找那个修了 3 次的死锁问题`
- 或给出具体的时间范围、文件名、commit hash 等线索

---

## Stage 1: COLLECT（自动收集证据）

### 1.1 定位相关 Commits

基于用户提供的线索（时间、关键词、文件名），自动搜索 git 历史：

```bash
# 按时间范围搜索
git log --all --oneline --since="<date>" --until="<date>" --format="%h %ad %s" --date=short

# 按关键词搜索
git log --all --oneline --grep="<keyword>" --format="%h %ad %s" --date=short

# 按文件名搜索
git log --all --oneline -- "<file-path>"
```

### 1.2 提取多轮修复 Commit

识别针对同一问题的多次修复，分组展示：

```bash
# 对每个定位到的 commit，查看完整 diff
git show <commit-hash> --stat
git show <commit-hash> --format="%H%n%ad%n%s%n%b" --date=short
```

**关键动作**：将疑似同一问题的多次修复 commit 分组，标注为 Round 1, Round 2, ... Round N。

### 1.3 构建时间线

```bash
# 按时间顺序列出所有相关 commit
git log --all --oneline --format="%h %ad %s" --date=short --reverse <commit-range>
```

### 1.4 输出 COLLECT 摘要

产出结构化的事实清单：
- **时间线**：从引入到最终修复的完整 commit 序列
- **涉及文件**：每个 round 修改了哪些文件
- **多轮修复链**：Round 1 → Round 2 → ... → Round N（最终修复）

---
### Gate
- ✅ 通过：至少定位到 2 轮修复 commit 且时间线清晰
- ❌ 阻塞：无法定位相关 commit、或只有一轮修复（不是 case-study 的适用场景）
---

## Stage 2: ANALYZE（根因分析）

### 2.1 逐轮对比分析（核心抓手）

对 Round 1 到 Round N 的每一次修复，逐轮分析：

```
Round 1:
  - 改了哪里？（精确到代码行）
  - 当时的假设是什么？（从 diff 推断）
  - 为什么不够？（漏掉了什么边界条件/场景）
  - 是否引入了新问题？

Round 2:
  - 补了什么？
  - 为什么还是不够？
  - 和 Round 1 之间有什么关联？

...

Round N（最终修复）：
  - 这次改了什么？和之前所有 round 的本质区别是什么？
  - 为什么这次终于对了？
```

### 2.2 跨轮归纳

从多轮修复中提取共性问题：
- **测试覆盖**：缺少哪些测试用例？如果有测试，哪一轮就该发现问题？
- **审查盲区**：code review 为什么没发现问题？是否缺少特定领域的 reviewer？
- **架构设计**：问题是否源于模块边界不清晰、依赖关系混乱？
- **拆分粒度**：是否一次修了太多东西？单次修复的复杂度是否过高？

### 2.3 5 Whys 追问

从表面现象开始，逐层追问：

```
Why 1: 为什么会出现这个事故？
  → 因为 [直接原因]

Why 2: 为什么 [直接原因] 会发生？
  → 因为 [更深层原因]

Why 3: 为什么 [深层原因] 没有被发现？
  → 因为 [流程/机制缺失]

Why 4: 为什么 [流程缺失] 存在？
  → 因为 [系统性原因]

Why 5: 为什么 [系统性原因] 没有被解决？
  → [根因]
```

### 2.4 影响范围

自动推断影响范围，仅在无法确定时向用户一次性确认：
- 影响用户数（如无法推断则询问）
- 影响时长（从 git 时间戳和部署记录推断）
- 影响数据量（从 diff 中的业务逻辑推断）

**提问规则**：只在无法从代码/git 推断时提问，且一次性问完所有不确定项。

---
### Gate
- ✅：逐轮分析完成 + 5 Whys 至少追到第 4 层 + 跨轮归纳有实质发现
---

### 3.3 self-review

生成后进行自检：
1. 所有 commit hash 正确（已从 git 输出中提取）
2. diff 代码片段真实（来自 `git show` 输出）
3. 5 Whys 至少追到第 5 层
4. 跨轮归纳有实质内容（非"下次注意"之类的空话）
5. 长期解决方案具体可执行（非"加强测试""完善文档"等笼统表述）

- ✅：文档完整生成 + self-review 通过

## Stage 4: DELIVER（交付）

### 4.1 输出摘要

呈现核心发现：
```
CASE STUDY 完成
===============
文件：docs/case-studies/{YYYYMMDD}-{slug}.md

核心发现：
  1. 共 {N} 轮修复才彻底解决
  2. 根因：{一句话根因}
  3. 关键教训：{最重要的一条教训}

长期行动项：
  - [ ] {action 1}
  - [ ] {action 2}
```

### 4.2 后续操作

不自动 commit/PR。用户可自行：
- 查看、修改文档
- `git add docs/case-studies/` 并 commit
- 如有需要，PR 供团队评审
